# Specification: Section Fragment Schema

## Authority

This file defines the logical SQL Response Engine section fragment schema consumed by the API section loop.

It decides fragment identity, ordering fields, decoded projection shape, reference rules, persistence rule, and commit success condition.

It does not define physical table DDL, temporary context table schema, object storage layout, or API memory implementation.

# Core Rule

A section fragment is a SQL-produced decoded context projection for API-side merge and reinjection.

It is not raw source context authority.

It is not semantic authority.

The SQL Response Engine owns:

```text
fragment production
res_context projection
source reference preservation
evidence reference preservation
SQL-side ordering metadata
```

The API Thinking Engine owns:

```text
tmp entry shaping
key retention
merge_order assignment
sort / merge
context reconstruction
reinjection
delivery after commit success
```

---

# Canonical Fragment Shape

SQL Response Engine section calls should return fragments shaped like:

```text
section_kind
section_run_id
res_key
sequence_tag
split_parent_token_index
split_position
res_context
source_refs
evidence_refs
state
```

This shape is logical.

Concrete implementations may return a composite type, `jsonb`, `SETOF` records, or a wrapper result as long as these fields remain available.

---

# Field Rules

## section_kind

Allowed values:

```text
split
think
generate
audit
```

`section_kind` identifies the API section request that produced the fragment.

It is orchestration lineage, not semantic authority.

## section_run_id

`section_run_id` is generated by SQL once per SQL section call.

It is a UUID because it identifies a run / event scope.

It must not be used as a semantic reference.

All fragments produced by the same committed section call share the same `section_run_id`.

Idempotent replay of the same committed section call must resolve the previously committed `section_run_id` rather than minting a new fragment set.

## res_key

`res_key` is the stable fragment key used by API-side temporary context storage and ordering.

Recommended form:

```text
<section_run_id>:<zero_padded_fragment_ordinal>
```

Example:

```text
7f6b8b30-19ec-4c5b-8df5-9e0ec2a94e13:000001
```

`res_key` is row / projection identity for runtime orchestration.

It is not a semantic reference key.

## sequence_tag

`sequence_tag` is a SQL-produced sortable text tag.

Canonical type:

```text
text
```

Recommended form:

```text
<section_kind>:<zero_padded_fragment_ordinal>
```

Examples:

```text
split:000001
think:000001
generate:000001
audit:000001
```

`sequence_tag` must be stable for the committed fragment set.

It must be lexicographically sortable.

It must not be derived from unordered JSON object keys.

It must not be treated as a semantic path.

## split_parent_token_index

`split_parent_token_index` preserves the parent token index when SQL length-based splitting creates child token rows or child decoded projections.

It is a generated numeric token index.

It must not be a UUID.

It may be null for fragments not produced from a split child.

## split_position

`split_position` preserves SQL-owned split order.

Canonical type:

```text
integer
```

It is zero-based or one-based according to implementation policy, but the policy must be consistent inside one committed fragment set.

It may be null for non-split fragments.

When both `sequence_tag` and `split_position` are present, `sequence_tag` is the primary ordering key and `split_position` is a tie-breaker / split provenance field.

For split fragments, SQL should derive `sequence_tag` from the same ordered split source as `split_position`.

A committed fragment set where `sequence_tag` and `split_position` imply contradictory order is invalid.

## res_context

`res_context` is SQL-produced decoded context projection.

Canonical type:

```text
jsonb
```

Recommended logical schema:

```json
{
  "projection_kind": "decoded_context",
  "section_kind": "split | think | generate | audit",
  "body": "decoded projection body or structured decoded projection",
  "summary": "optional compact projection summary",
  "decode_metadata": {
    "decoder_kind": "optional decoder classification",
    "budget_applied": {},
    "truncated": false
  },
  "lineage": {
    "section_run_id": "uuid string",
    "res_key": "stable fragment key",
    "parent_tmp_context_key": "optional API-side parent key"
  },
  "warnings": []
}
```

`res_context` may contain decoded text or structured decoded projection needed for API merge.

It must not replace canonical reply-time structures.

It must not be used as semantic authority.

It must not carry UUID arrays as semantic paths.

Large raw reply context bodies should remain in committed reply-core storage or referenced storage, not in `res_context` as authority.

## source_refs

`source_refs` preserves links back to committed reply-core structures and temporary projection lineage.

Canonical type:

```text
jsonb
```

Recommended logical schema:

```json
{
  "input_observation_index": 0,
  "token_path": [],
  "vocabulary_path": [],
  "grammar_path": [],
  "grammar_relation_path": [],
  "split_parent_token_index": null,
  "tmp_context_key": null,
  "ordered_decode_keys": []
}
```

Semantic references inside `source_refs` must use generated numeric indexes or `bigint[]` index paths.

UUIDs may identify rows, events, or run scopes only.

## evidence_refs

`evidence_refs` preserves committed evidence references used by the fragment.

Canonical type:

```text
jsonb
```

Recommended logical schema:

```json
{
  "coherence_log_indexes": [],
  "diff_log_indexes": [],
  "aggregate_refs": [],
  "scheduler_job_run_refs": [],
  "archive_indexes": []
}
```

Evidence references should use generated indexes and routed reference objects.

Evidence UUIDs may appear only as row / event identity inside evidence JSON when required for traceability.

Logs record evidence.

Logs do not become semantic authority.

## state

`state` is fragment delivery state.

Recommended values:

```text
produced
empty
truncated
blocked
error
```

`state` is not semantic lifecycle authority.

It must not be used to promote, delete, adopt, or freeze semantic structures.

Semantic lifecycle truth remains with canonical flags and operation-gated log evidence.

# Ordering Rule

API reconstruction must sort by stable ordering metadata.

Canonical reconstruction priority:

```text
section_order
merge_order
sequence_tag
split_position
res_key
```

`section_order` is the API section order:

```text
split -> think -> generate -> audit
```

`merge_order` may be assigned by the API when shaping fragments into temporary context.

`sequence_tag`, `split_position`, and `res_key` are SQL-produced fragment ordering fields.

The API must not infer order from unordered JSON object keys.

---

# Persistence Rule

A section fragment may be:

```text
returned directly from the SQL function
written as a decoded projection to temporary context storage
represented by a stable key pointing to an external decoded projection body
```

The logical fragment boundary is the same regardless of storage form.

Temporary context entries may add:

```text
tmp_context_key
merge_order
parent_tmp_context_key
created_at
```

The API may store and merge decoded projections through temporary context.

Temporary context storage remains decoded projection / orchestration state only.

It must not replace canonical reply-time structures, logs, or semantic authority.

---

# Commit Success Condition

A section fragment set is API-usable only when all of the following are true:

```text
all required canonical source refs resolve to committed rows
all required evidence refs resolve to committed rows
fragment ordering metadata is stable
runtime result envelope is produced for the same route
SQL transaction commits successfully
```

If the transaction rolls back, the fragments, ordered decode keys, decoded projections, and envelope are invalid.

If a route intentionally produces partial output, it must still be a committed partial result with a committed envelope.

`partial_output` means committed partial candidate, not uncommitted stream leakage.

---

# Reinjection Rule

A reconstructed decoded context may be reinjected into a later section only as orchestration input.

Reinjected context must carry:

```text
parent_tmp_context_key
ordered_decode_keys
section lineage
source_refs
evidence_refs
```

Reinjection is not semantic verification, promotion, mutation authority, or lifecycle authority.

---

# Prohibited Patterns

```text
No raw source context as fragment authority.
No res_context as semantic authority.
No tmp_context as semantic authority.
No UUID semantic arrays.
No UUID semantic path components.
No status enum lifecycle authority.
No ordering from unordered JSON object keys.
No API-side rewrite of source_refs or evidence_refs as semantic proof.
No scheduler job promotion authority.
No adopt.* operation keys.
```
